# Workspace UI 与卡片系统对齐同步设计

日期：2026-04-24

## 背景

当前前端已经完成医生端和患者端的新视觉框架：深色顶栏、三栏工作区、右上角头像旁角色切换、中文界面和默认空态。与此同时，原本前端已有一批必须保留的联动能力：SSE 对话流、上传资料、患者库绑定、卡片 upsert、inline cards、工作流路线、执行计划、引用列表、患者身份填写和重置场景。

这次对齐同步的目标不是再做一套页面，而是把原有功能内核接回新视觉壳层，确保新界面只展示真实状态，不展示固定样本。

## 目标

1. 医生端和患者端统一使用新的视觉框架，但不丢失原本联动功能。
2. 卡片、流程、引用、上传和患者摘要全部从会话状态或后端响应派生。
3. 默认界面和重置后界面必须是真空态，不出现 `P-1024`、`CT Report`、固定治疗建议等演示内容。
4. 医生/患者切换固定在右上角头像旁，场景切换不再作为独立业务按钮出现在顶栏中段。
5. 保留卡片系统的原渲染能力，新增视觉层只负责布局、样式、空态和高亮。

## 非目标

1. 不重写后端 API。
2. 不重构所有历史测试或数据库工作台。
3. 不把参考图内容写成静态页面。
4. 不在真实工作区启用默认演示卡片。
5. 不改变现有对话流、上传流、患者库绑定流的接口契约。

## 推荐方案

采用“功能内核不动，视觉壳层收敛”的渐进式方案。

`WorkspacePage` 继续作为总编排层，负责 scene、session、SSE、上传、reset、latency、cards 合并和错误处理。医生端与患者端组件只消费 props，不主动创建业务数据。卡片系统保留原有 `renderCardContent` 和 `card-renderers-extended`，在外层增加统一的临床卡片展示适配规则。

这个方案风险最低，因为它避免重写已有联动逻辑，同时能逐步消除当前出现的两类问题：新 UI 中残留演示数据，以及旧功能只能通过隐藏兼容入口触发。

## 架构边界

### WorkspacePage

职责：

- 管理 `patient` 和 `doctor` 两个 scene 的会话状态。
- 从 `SessionState` 派生对话、卡片、流程、执行计划、引用、上传状态和患者身份。
- 处理发送消息、上传文件、重置场景、切换场景、绑定患者和延迟统计。
- 将派生后的数据传给医生端和患者端视图。

约束：

- 不写入演示业务内容。
- 不在真实 workspace 中启用 `showDefaultCards`。
- 重置遇到 stale session 时创建新会话，并替换旧状态。

### DoctorSceneShell

职责：

- 渲染医生端顶栏、三栏布局、患者摘要、上传摘要、对话、医疗卡片、路线图、执行计划、引用列表和事件流。
- 通过右上角头像旁入口切换到患者端。
- 医生端 tab 只控制医生工作区内部视图，例如会诊和患者数据库。

约束：

- 患者摘要只来自绑定患者详情或真实 `patient_card`。
- 上传摘要只来自患者记录或真实 uploaded assets。
- 无数据时展示空态，不补样本。

### Patient Scene

职责：

- 渲染患者端顶栏、患者信息、患者背景、对话和上传资料。
- 通过右上角头像旁入口切回医生端。
- 患者背景只展示真实 `patient_card` 中已确认字段。

约束：

- 不展示参考图中的固定症状清单、流程节点或上传文件。
- `资料填写 / 症状 / 上传 / 照护计划` 这些 tab 在没有真实路由前只作为导航状态或禁用入口，不模拟业务结果。

### ClinicalCardsPanel

职责：

- 负责临床卡片区域的视觉呈现：标题、网格、色调、空态、闪烁高亮和卡片容器。
- 将每张真实卡片交给 `renderCardContent` 渲染内容。

约束：

- 默认真实 workspace 中 `cards={}` 时只展示空态。
- `showDefaultCards` 只允许用于隔离 demo 或测试。
- 卡片标题、内容和操作均来自真实 payload 或 renderer，不由面板硬编码业务结论。

## 数据流

1. 后端返回 `SessionResponse`。
2. `hydrateSessionState` 写入 `SessionState`。
3. `WorkspacePage` 先用 `cardsWithInlineCards` 合并会话卡片和消息内联卡片。
4. `getVisibleCards` 根据当前场景和 triage 状态过滤卡片。
5. 医生端和患者端分别消费过滤后的卡片。
6. `ClinicalCardsPanel` 只负责容器视觉，实际卡片内容交给 card renderer。

患者摘要数据优先级：

1. 医生端绑定患者详情。
2. 当前会话中的 `patient_card`。
3. 空态。

患者背景数据优先级：

1. 患者端真实 `patient_card`。
2. 空态。

上传数据优先级：

1. `uploadedAssets` 或患者记录。
2. 空态。

## UI 对齐规则

1. 顶栏风格统一使用 `clinical-top-nav`。
2. 医生端和患者端右上角都使用 `clinical-profile-switch` 作为场景切换入口。
3. 内容区统一使用浅灰背景、白色 8px 面板、三栏布局和清晰空态。
4. 卡片区域不使用营销式大卡或固定 hero 内容。
5. 空态文案必须短，不暗示已有样本数据。
6. 患者端保持和医生端相同的留白密度，不按参考图填满固定内容。

## 兼容入口清理

当前医生端仍保留隐藏的兼容控制，用于让旧测试和旧入口还能触发会诊/患者库。后续应逐步清理：

1. 将测试改为直接点击新顶栏按钮。
2. 确认患者库、会诊工作台、通用临床模式都能通过新顶栏触发。
3. 删除隐藏兼容 DOM。
4. 删除对应的隐藏 CSS。

这个清理必须在测试覆盖完成后进行，避免破坏患者库和会诊联动。

## 错误处理

1. `resetSession` 返回 404 时，视为本地持久化 session 已过期。
2. 前端调用 `createSession(activeScene)` 创建新场景会话。
3. 新响应必须覆盖旧场景状态，并清空 draft、latency 和上传状态。
4. 非 404 错误继续展示错误消息，不清除现有状态。
5. 上传 413 继续映射为用户友好的文件大小错误。

## 测试策略

新增或保持以下测试面：

1. 初始空态：医生端和患者端均不出现演示样本。
2. 重置空态：重置后不出现旧患者摘要、旧上传项或旧卡片。
3. stale reset：`resetSession` 404 后会创建新当前场景会话。
4. 场景切换：右上角头像旁入口可以医生和患者互切。
5. 卡片联动：message inline cards 能进入可见卡片区域。
6. 患者背景：只展示真实 `patient_card` 字段，空 scaffold 显示空态。
7. 卡片面板：默认空态；只有显式 `showDefaultCards` 才展示 demo cards。

## 实施顺序

1. 建立功能矩阵，逐项标记旧前端能力在新 UI 中的入口和数据来源。
2. 统一医生端和患者端的数据派生，清除真实 workspace 中的默认演示内容。
3. 明确卡片 view model 或适配函数，减少 UI 组件读取 payload 细节。
4. 更新测试从新顶栏入口触发交互。
5. 删除隐藏兼容控制。
6. 回归浏览器验证医生端空态、患者端空态、对话、上传、患者库绑定和卡片联动。

## 验收标准

1. 新启动或强制刷新后，医生端默认是空态。
2. 患者端默认是空态，只显示真实患者信息、背景、对话和上传状态。
3. 对话返回卡片后，卡片区域同步更新。
4. 上传资料后，上传区域和患者背景相关卡片同步更新。
5. 患者库绑定后，医生端患者摘要同步更新。
6. `npm --prefix frontend run test -- --run` 通过。
7. `npm --prefix frontend run build` 通过。
8. 浏览器中右上角角色入口可以双向切换。

## 风险与缓解

风险：删除隐藏兼容入口可能破坏旧测试或旧键盘路径。  
缓解：先把测试迁移到新顶栏入口，再删除兼容 DOM。

风险：卡片 payload 类型不一致导致患者背景或医生摘要显示错误。  
缓解：新增卡片适配层，集中做字段读取和空态判断。

风险：demo cards 再次进入真实 workspace。  
缓解：测试断言真实 workspace 默认不渲染 demo card 文案。

风险：stale session 导致旧状态残留。  
缓解：404 reset fallback 已纳入设计和测试范围，需保持结构化 404 判断。

## 决策

采用渐进式对齐方案。当前阶段优先保证真实数据同源、空态正确、角色切换入口统一；随后再抽取共享布局和卡片 view model，降低长期维护成本。
